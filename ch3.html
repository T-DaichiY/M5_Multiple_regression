Chapter 3

⚖️ 標準化回帰係数

湿度・気温・風速は単位がバラバラ。係数の大きさをそのまま比べても「どれが一番効いているか」はわからない。単位をそろえて公平に比べる方法を学ぶ。

①

なぜ係数をそのまま比べられないのか

単位が違うとリンゴとオレンジを比べることになる

前章のモデルでは、湿度・気温・風速でレンタル数を予測しました。でも係数をそのまま比べるのはちょっと厄介です。なぜなら単位（スケール）が違うから。風速・湿度・気温はそれぞれ別の物差しで測られています。

⚠️ よくある勘違い：「湿度の係数が −3100 で一番大きいから、湿度が一番重要だ！」── これは間違いです。係数の大きさは単位に左右されます。しかも、すべての予測変数のp値はとても小さい（みんな有意）。では本当にどれが一番効いているのでしょう？
単位が違うと比べられない
🍎 かんたん説明

身長を「センチ」、体重を「キロ」で測って「数字が大きいから身長のほうが大事」とは言えませんよね。単位が違うものはそのまま比べられない。だからみんな同じ物差しにそろえてから比べます📏

②

方法1：手動でZスコア化

仕組みを理解するための長いやり方

3つの方法を紹介します（おすすめは方法3）。まず仕組みを理解するため、手動で標準化（Zスコア化）してみましょう。Zスコアの計算は「観測値 − 平均 ÷ 標準偏差」です。

# Zスコア = (値 - 平均) / 標準偏差bikes_for_beta <- bikes %>% mutate(z_rentals = (rentals - mean(rentals)) / sd(rentals), z_humidity = (humidity - mean(humidity)) / sd(humidity), z_temp = (temperature - mean(temperature)) / sd(temperature), z_wind = (windspeed - mean(windspeed)) / sd(windspeed))

たとえばレンタル985件の日のZスコアが −1.8 なら、「平均より約2標準偏差ぶん下」という意味です。標準化したデータで重回帰を作ると、新しい係数が得られます。

model_std <- lm(z_rentals ~ z_humidity + z_temp + z_wind, data = bikes_for_beta)
標準化後の係数（絶対値で比較）: z_temp ← 一番大きい！ = 気温が最も影響大 z_humidity z_wind

符号は無視して絶対値（大きさ）を見ると、気温（temperature）が一番大きい。つまり気温がレンタル数に最も影響している、とわかります。これは相関（レンタル数と最も強く相関するのも気温）やt統計量（summaryの3列目、気温が最大の絶対値）でも確認できます。

🍎 かんたん説明

Zスコアは「テストの偏差値」みたいなもの。点数(単位)が違う英語と数学のテストも、偏差値にすれば「どっちが得意か」を公平に比べられますよね。標準化＝みんなを偏差値にそろえる作業です📝

③

方法2：scale関数（もっと簡潔に）

Zスコア化を一発でやってくれる

手動でmutateするのは大変。scale() 関数はZスコア化を自動でやってくれます。前の長いコードがぐっと短くなります。

bikes_for_beta2 <- bikes %>% select(rentals, humidity, temperature, windspeed) %>% as.data.frame() %>% scale() # 全列を一気にZスコア化model_std2 <- lm(rentals ~ humidity + temperature + windspeed, data = bikes_for_beta2)# → 方法1と全く同じ係数になる
📌 結果は同じ：手動Zスコアでも scale() でも、得られる標準化係数は同じです。scale() のほうがコードが短くてミスも少ないので便利です。
🍎 かんたん説明

方法1が「全部手書きで偏差値を計算」なら、方法2は「計算機のボタン1つで偏差値表を作る」。同じ答えなら、ラクなほうがいいですよね😊

④

方法3：lm.beta パッケージ（一番おすすめ）

データを変換せず、モデルを渡すだけ

もっと効率的なのが lm.beta パッケージ。データそのものを標準化する必要がなく、作ったモデルを渡すだけで標準化係数が得られます。

# install.packages("lm.beta") が必要library(lm.beta)lm.beta(bikes_model_1) # 標準化係数を表示（気温が最大）summary(lm.beta(bikes_model_1)) # 元の係数＋標準化列を併記

summary(lm.beta(...)) は、元の推定値と標準化された列（Standardized）を並べて表示してくれるので便利です。

🔢 コードグレード対策：「最良の予測変数の標準化係数」だけを取り出すには：
summary(lm.beta(bikes_model_1))$coefficients[3, 2]# 3行目(気温)・2列目(標準化係数)を抜き出す
⚠️ "beta" という言葉に注意：「beta」は多義語で混乱しがちです。① 標準化回帰係数の記号、② 一部の教科書では非標準化係数 β₀,β₁…、③ 母集団の回帰直線の傾き、④ 統計的検出力の話でも登場。課題では「標準化回帰係数」または「lm.beta」と明示します。
🍎 かんたん説明

方法3は「魔法の翻訳メガネ」。データをいじらなくても、メガネをかける（lm.betaに通す）だけで、バラバラの単位が一瞬で「同じ物差し」に見える！一番ラクで一番おすすめ🪄

⑤

解釈と注意点

便利だけど、使いどころを間違えないこと

どう解釈する？

標準化係数で β₁＝0.5、β₂＝1 なら、「X₂はX₁の2倍、Yの予測に重要」と言えます。ただし前提として、両者がだいたい同じ分布で、標準偏差が大きく違わないことが必要です。

👍 メリット

  • 元々スケールが違う予測変数の相対的な重要度を直接比較できる。
  • 変数の重要度をすばやく把握でき、データ探索に役立つ。
  • 同じ物差しなので、非専門家にも説明しやすい。

👎 注意点（カウション）

  • 実用的な意味が失われることもある。「登録1件あたり$100」のほうが「0.2標準偏差」より直感的（特に非専門家には）。
  • 予測がやや複雑になる（標準化を戻す必要がある）。
  • 変数が同じ分布・似た標準偏差である前提が、現実には大きな仮定になりうる。
  • 標準化係数はサンプルのばらつきに影響され、他のデータセットへの一般化がしにくい。
📌 おすすめ：聞き手に応じて、元の係数と標準化係数の両方を報告しましょう。状況によって両方必要になります。
🍎 かんたん説明

偏差値は「学校内での順位」を比べるのに便利だけど、「実際に何点だったか」も大事ですよね。標準化係数（偏差値）と元の係数（実際の点数）、両方を見せるのが親切なやり方です🎓
